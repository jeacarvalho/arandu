### Padrões Aplicados (Regras de Ouro)

✅ **Independência de Domínio**
- Handlers não contêm lógica de negócio
- Apenas: 1. Decodificam Request → 2. Chamam Service → 3. Mapeiam ViewModel → 4. Renderizam Template

✅ **Consciência de Contexto HTMX**
- Cada handler verifica `HX-Request` header
- Se HTMX: renderiza apenas fragmento (`patient-content`, `session-content`)
- Se não HTMX: renderiza página completa com `layout.html`

✅ **Tipagem Forte (ViewModels)**
- Nunca passa entidades de domínio diretamente para templates
- ViewModels específicos garantem que template tenha exatamente o que precisa
- Exemplo: `PatientViewModel` tem campos formatados para exibição (data formatada, etc.)

✅ **Injeção de Dependência**
- Interfaces para todos serviços (PatientService, SessionService, etc.)
- Facilita testes unitários e mocking
- Segregação de interfaces (cada handler define apenas o que usa)

✅ **Tratamento de Erros**
- Fragmentos de erro amigáveis para HTMX
- Status codes apropriados (400, 404, 500)
- Mensagens claras para usuário final

✅ **DRY Templates**
- Uso de `{{template}}` para evitar duplicação
- `layout.html` como esqueleto único
- Fragmentos nomeados especificamente (`patient-content`, `session-content`)

### Problemas Resolvidos

1. **Layout duplicado** - Anteriormente alguns templates tinham `<html>` completo
   - Solução: Todos usam `{{define "content"}}` injetado no `layout.html`

2. **Conflito de nomes de templates** - Template "content" genérico causava sobrescrita
   - Solução: Nomes específicos (`patient-content`, `session-content`, `patients-content`)

3. **Entidades de domínio expostas** - Templates acessavam campos internos
   - Solução: ViewModels intermediários com campos explícitos

4. **Erros 500 brutos no HTMX** - Fragmentos recebiam HTML de erro puro
   - Solução: `renderError()` com fragmento amigável `error-fragment`

5. **Handlers com lógica de negócio** - Validações e regras misturadas
   - Solução: Handlers apenas orquestram, serviços têm lógica

### Prevenção Futura

1. **Sempre criar ViewModel** antes de passar dados para template
2. **Sempre verificar HX-Request** em handlers que podem ser chamados via HTMX
3. **Usar nomes únicos** para templates fragmento (evitar "content" genérico)
4. **Testar ambos cenários**: requisição direta e via HTMX
5. **Manter handlers finos** - se tiver lógica complexa, mover para service
6. **Documentar ViewModels** - explicar propósito de cada campo

### Lições Aprendidas

1. **Clean Architecture na prática**: A separação clara entre handlers (web), services (aplicação) e domain (core) facilita manutenção e testes

2. **HTMX requer consciência contextual**: Não é só "retornar HTML", é retornar o HTML certo (fragmento vs página completa) baseado no contexto da requisição

3. **ViewModels são proteção**: Eles protegem o domínio de mudanças na UI e vice-versa. Se o template precisar de um campo formatado, o ViewModel faz isso sem poluir a entidade de domínio

4. **Interfaces específicas > Interfaces genéricas**: Cada handler define exatamente as interfaces que usa, facilitando testes e reduzindo acoplamento

5. **Erro amigável é feature**: Em aplicações HTMX, erros devem ser fragmentos HTML úteis, não códigos de status brutos
